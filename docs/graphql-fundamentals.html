üìö What is GraphQL?

Simple Definition

GraphQL is a query language for APIs that allows you to ask for exactly what you need, nothing more, nothing less.

üéØ Real-Life Analogy

Restaurant Example:

REST API: Like ordering a set meal - you get everything whether you want it or not (rice, dal, curry, roti, salad)

GraphQL: Like ordering √† la carte - you choose exactly what you want (just rice and dal, or just curry and roti)

üìñ History

  • Created by Facebook in 2012
  • Made open source in 2015
  • Now used by: Facebook, GitHub, Twitter, Shopify, Netflix, Airbnb

üî§ GraphQL Basics

1. Single Endpoint Architecture

Unlike REST which has multiple URLs, GraphQL uses ONE endpoint for everything.

REST (Multiple Endpoints):

GET  /api/students           ‚Üí Get all students
GET  /api/students/1         ‚Üí Get student by ID
POST /api/students           ‚Üí Create student
GET  /api/departments        ‚Üí Get departments
GET  /api/students/1/marks   ‚Üí Get student marks

GraphQL (Single Endpoint):

POST /graphql  ‚Üí ALL operations (queries, mutations)

2. Three Main Operations

Operation Purpose Similar to REST Example
Query Read/Fetch data GET Get student details
Mutation Create/Update/Delete POST, PUT, DELETE Add new student
Subscription Real-time updates WebSockets Live notifications

3. Basic Query Structure

query {
  operationName {
    field1
    field2
    field3
  }
}

Simple Example:

query {
  student {
    name      ‚Üê Want this
    email     ‚Üê Want this
    age       ‚Üê Want this
  }
}

Response:
{
  "data": {
    "student": {
      "name": "Ravi Kumar",
      "email": "ravi@example.com",
      "age": 20
    }
  }
}

4. Basic Mutation Structure

mutation {
  operationName(argument1: value1, argument2: value2) {
    field1
    field2
  }
}

Simple Example:

mutation {
  createStudent(name: "Priya", email: "priya@example.com") {
    id          ‚Üê Return the new ID
    name        ‚Üê Return the name
    email       ‚Üê Return the email
  }
}

Response:
{
  "data": {
    "createStudent": {
      "id": "123",
      "name": "Priya",
      "email": "priya@example.com"
    }
  }
}

üí° Core Concepts

1. Field Specification

Field Specification means you choose exactly which fields you want in the response.

Example: Student Data

Available Fields:
- id
- name
- email
- age
- phone
- address
- marks
- department

You ask for only:
query {
  student {
    name        ‚Üê Only these 3 fields
    email
    age
  }
}

You get only:
{
  "name": "Ravi",
  "email": "ravi@example.com",
  "age": 20
}

2. Graph-Based Data Structure

Graph-Based means data is connected like a graph. You can follow relationships to get related data in one request.

Fetching Related Data:

query {
  student {
    name
    email
    department {          ‚Üê Follow relationship
      name
      hod
      building
    }
    marks {              ‚Üê Another relationship
      subject
      score
    }
  }
}

Response - Everything in one request!
{
  "student": {
    "name": "Ravi",
    "email": "ravi@example.com",
    "department": {
      "name": "BCA",
      "hod": "Dr. Rama",
      "building": "Block A"
    },
    "marks": [
      { "subject": "Python", "score": 85 },
      { "subject": "JavaScript", "score": 90 }
    ]
  }
}

3. Strongly Typed System

Strongly Typed means every field has a specific data type (like String, Int, Boolean). This helps catch errors early!

Data Types in GraphQL:

Type Description Example Values
String Text data "Ravi Kumar", "BCA"
Int Whole numbers 20, 100, -5
Float Decimal numbers 8.5, 3.14, 99.9
Boolean True or False true, false
ID Unique identifier "abc123", "user_001"
! Required (non-null) String! = must have value
[ ] List/Array [Student] = list of students

4. Introspection

Introspection means GraphQL can describe itself. You can ask "What queries are available?" or "What fields does Student have?"

Ask GraphQL about itself:

Query 1: What queries are available?
query {
  __schema {
    queryType {
      fields {
        name
        description
      }
    }
  }
}

Query 2: What fields does Student type have?
query {
  __type(name: "Student") {
    fields {
      name
      type {
        name
      }
    }
  }
}

Response:
{
  "__type": {
    "fields": [
      { "name": "id", "type": { "name": "ID" } },
      { "name": "name", "type": { "name": "String" } },
      { "name": "email", "type": { "name": "String" } },
      { "name": "age", "type": { "name": "Int" } }
    ]
  }
}

üí° Why Introspection is Awesome:

  • Tools like GraphiQL use it for auto-complete
  • Automatically generate documentation
  • Build smart IDE plugins
  • No need for separate API docs!

üìù Types & Schema

1. What is a Schema?

A Schema is like a blueprint or contract. It defines:

  • What data is available
  • What operations you can perform
  • What types of data exist

2. Object Types

Student Type Definition:

type Student {
  id: ID!              ‚Üê ID type, Required (!)
  name: String!        ‚Üê String type, Required
  email: String!       ‚Üê String type, Required
  age: Int             ‚Üê Integer type, Optional
  phone: String        ‚Üê String type, Optional
  isActive: Boolean    ‚Üê Boolean type, Optional
  gpa: Float          ‚Üê Float type, Optional
}

Department Type Definition:

type Department {
  id: ID!
  name: String!
  code: String!
  hod: String!
  building: String
  students: [Student]   ‚Üê List of Student objects
}

3. Relationships Between Types

type Student {
  id: ID!
  name: String!
  department: Department   ‚Üê Student belongs to Department
}

type Department {
  id: ID!
  name: String!
  students: [Student]      ‚Üê Department has many Students
}

4. Query Type

Query Type defines all READ operations available in your API.

type Query {
  # Get single student by ID
  student(id: ID!): Student
  
  # Get all students
  students: [Student]
  
  # Search students by name
  searchStudents(name: String!): [Student]
  
  # Get single department
  department(id: ID!): Department
  
  # Get all departments
  departments: [Department]
}

5. Mutation Type

Mutation Type defines all CREATE, UPDATE, DELETE operations.

type Mutation {
  # Create new student
  createStudent(
    name: String!
    email: String!
    age: Int
    departmentId: ID!
  ): Student
  
  # Update existing student
  updateStudent(
    id: ID!
    name: String
    email: String
    age: Int
  ): Student
  
  # Delete student
  deleteStudent(id: ID!): Student
  
  # Create department
  createDepartment(
    name: String!
    code: String!
    hod: String!
  ): Department
}

6. Input Types

Input Types are used to pass complex objects as arguments.

# Define Input Type
input StudentInput {
  name: String!
  email: String!
  age: Int
  phone: String
  departmentId: ID!
}

# Use in Mutation
type Mutation {
  createStudent(input: StudentInput!): Student
}

# Usage
mutation {
  createStudent(input: {
    name: "Ravi"
    email: "ravi@example.com"
    age: 20
    departmentId: "123"
  }) {
    id
    name
  }
}

7. Custom Scalar Types

Custom Scalars let you define your own data types.

Common Custom Scalars:

scalar Date          # For dates: "2024-01-15"
scalar DateTime      # For date + time
scalar Email         # Validates email format
scalar URL           # Validates URL format
scalar PhoneNumber   # Validates phone numbers

type Student {
  id: ID!
  name: String!
  email: Email         ‚Üê Custom Email type
  birthDate: Date      ‚Üê Custom Date type
  website: URL         ‚Üê Custom URL type
}

üéØ Advanced GraphQL Concepts

1. Directives

Directives are special instructions that modify query execution.

@include Directive:

query GetStudent($withDepartment: Boolean!) {
  student(id: "123") {
    name
    email
    department @include(if: $withDepartment) {
      name
      code
    }
  }
}

# Variables
{ "withDepartment": true }   ‚Üê Include department
{ "withDepartment": false }  ‚Üê Don't include department

@skip Directive:

query GetStudent($skipAge: Boolean!) {
  student(id: "123") {
    name
    email
    age @skip(if: $skipAge)
  }
}

# Variables
{ "skipAge": true }   ‚Üê Skip age field
{ "skipAge": false }  ‚Üê Include age field

2. Pagination

Pagination helps load large datasets in chunks.

Offset-based Pagination:

query {
  students(limit: 10, offset: 0) {    ‚Üê First 10 students
    id
    name
  }
}

query {
  students(limit: 10, offset: 10) {   ‚Üê Next 10 students
    id
    name
  }
}

Cursor-based Pagination:

query {
  students(first: 10) {
    edges {
      node {
        id
        name
      }
      cursor
    }
    pageInfo {
      hasNextPage
      endCursor
    }
  }
}

When to Use What?

‚úÖ Use REST when:

  • Simple CRUD operations
  • Need HTTP caching
  • File uploads/downloads are primary
  • Team is familiar with REST
  • Small, simple applications

‚úÖ Use GraphQL when:

  • Complex data relationships
  • Multiple client apps (web, mobile, etc.)
  • Need flexibility in data fetching
  • Want to minimize network requests
  • Real-time features needed
  • Large, complex applications

üìù Quick Quiz

Question 1:

What is the main advantage of GraphQL over REST?

Show Answer

Answer: You can request exactly the data you need, avoiding over-fetching and under-fetching. You can also get related data in a single request instead of multiple requests.

Question 2:

What's the difference between a Query and a Mutation?

Show Answer

Answer: Queries are for reading/fetching data (like GET in REST). Mutations are for creating, updating, or deleting data (like POST, PUT, DELETE in REST).

Question 3:

What does the exclamation mark (!) mean in GraphQL types?

Show Answer

Answer: It means the field is required (non-null). For example, String! means this field must have a value and cannot be null.

Question 4:

How many endpoints does a GraphQL API typically have?

Show Answer

Answer: Just ONE endpoint (usually /graphql). All queries and mutations go to this single endpoint.

Question 5:

What is introspection in GraphQL?

Show Answer

Answer: Introspection is GraphQL's ability to describe itself. You can query the schema to find out what queries, mutations, and types are available, making GraphQL APIs self-documenting.
